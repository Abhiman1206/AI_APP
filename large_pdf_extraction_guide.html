Extracting Large PDFs
300–400 Pages

A complete procedure guide for your Intelli-Credit data ingestion pipeline — covering page routing, chunking strategy, tools, and production architecture.

01

Why Large PDFs Are Hard

Problem 1
Multi-column Layouts
Financial tables span 2–3 columns. Simple text extraction reads column 1 entirely before column 2, destroying table structure.
Problem 2
Mixed Content Types
Page 1 = text narrative. Page 45 = scanned image. Page 201 = dense financial table. No single tool handles all three well.
Problem 3
Context Window Limits
350 pages ≈ 200,000+ tokens. LLMs can't ingest this at once. Naive truncation loses critical financial data.
Problem 4
Silent Failures
Incorrect numeric extraction — e.g. ₹1,92,000 read as ₹192,000 — fails silently. Wrong CAM = wrong credit decision.
02

The 6-Stage Extraction Pipeline

📥
Stage 1
Ingest & Pre-flight Check
Before any extraction, scan the PDF to detect its type: is it text-native, scanned/image-based, or hybrid? Also detect total pages, language, and presence of text layer.
pypdf metadata pdfplumber.open() ~0.5s per PDF
🗺️
Stage 2
Intelligent Page Router
Classify each page (or sampled pages) into: plain text, multi-column text, financial table, chart/image, or scanned image. Route each to the right extractor. Don't send everything to one tool.
PyMuPDF → plain text pages pdfplumber → table pages Docling → complex layout pages PaddleOCR → scanned pages
✂️
Stage 3
Section Detection & Chunking
Use TOC (Table of Contents) or heading detection to divide the PDF into logical sections BEFORE chunking. For annual reports: Company Overview → Director's Report → Financial Statements → Notes to Accounts. Chunk within sections, not across them.
page-level chunks → best for finance ~800–2000 tokens per chunk 100-token overlap between chunks
🧠
Stage 4
Targeted Extraction (Priority Pages)
For the CAM, you don't need all 350 pages. Locate the ~30 high-value pages (Balance Sheet, P&L, Cash Flow, MD&A, Auditor's Report, Notes) and apply deep extraction only to those. Scan rest for qualitative signals.
~8% of pages = 80% of credit value Deep parse: 30 pages Keyword scan: 320 pages
✅
Stage 5
LLM Validation (Double-Check)
Send extracted financial figures to Claude API for cross-verification. Ask: "Does Total Assets = Total Liabilities + Equity?" and "Is PAT consistent across Balance Sheet and P&L?" LLM catches silent extraction errors before they reach your CAM.
Claude Sonnet 4.6 API accounting identity checks LLMChallenge pattern
💾
Stage 6
Structured Output + Vector Store
Write verified data to JSON (structured financials) + embed text chunks into a vector DB (FAISS/Pinecone) for RAG queries. This lets the Research Agent query "litigation mentions" or "promoter risk" across all ingested documents.
JSON → financial facts FAISS → qualitative RAG ready for CAM generator

Intelligent Page Router — Which Tool for Which Page?

Routing Decision Logic

Page Type
Plain Text Narrative
→ PyMuPDF / pdfplumber
MD&A, Director's Report, Chairman's Message. Fast (~40ms/page). Handles single-column well.
Page Type
Financial Table
→ pdfplumber + Docling
Balance Sheet, P&L, Cash Flow, Notes. Needs row-column relationship preservation. Docling: 97.9% table accuracy.
Page Type
Multi-column Layout
→ Docling MCP
Two-column financial statements (like Vivriti's page 201). Layout-aware AI prevents column bleed.
Page Type
Scanned / Image PDF
→ PaddleOCR MCP
Older Indian annual reports often have scanned pages. PaddleOCR supports 100+ languages including Hindi text in tables.
Page Type
Charts / Graphs
→ Skip or Vision LLM
Pie charts, bar graphs. Skip for structured extraction. If needed, pass as image to Claude vision for description.
Page Type
Filler Pages
→ Skip entirely
Cover pages, photo spreads, CSR imagery, blank pages. Detect by low text density (<50 chars) and skip.
def classify_page(page) → str: # Quick heuristic classifier — runs per page text = page.extract_text() or "" tables = page.extract_tables() chars = len(text.strip()) if chars < 50: return "filler" # skip if len(tables) > 0 and chars > 200: return "financial_table" # → Docling if any(kw in text for kw in ["Balance Sheet", "Profit and Loss", "Cash Flow"]): return "priority_financial" # → deep extraction if chars > 500: return "narrative_text" # → PyMuPDF fast path return "scan_candidate" # → PaddleOCR

Chunking Strategy for Financial PDFs

🔬 Research Finding (NVIDIA, 2025) Page-level chunking achieved the highest average accuracy (0.648) across all financial datasets — outperforming both token-based and section-level chunking. For annual reports: use page-level as your default.
STRATEGY 1 · RECOMMENDED

Page-Level Chunking

Each PDF page = one chunk. Works best for financial reports because financial statements are page-bounded. Preserves tables intact.

✅ Best for CAM
STRATEGY 2 · GOOD FOR MD&A

Section-Level Chunking

Use TOC to split at section boundaries. "Financial Statements" = one chunk. "Director's Report" = another. Great for qualitative RAG.

👍 Good for qualitative
STRATEGY 3 · HYBRID

~2000 Tokens + Overlap

Split at 2000 tokens with 100-token overlap. Never split mid-sentence. Use for Notes to Accounts which span multiple pages.

👍 Good for notes
STRATEGY 4 · AVOID

Fixed-Size Token Splitting

Splits blindly at N tokens regardless of content. Cuts financial tables in half. Destroys accounting relationships. Never use for structured financial data.

⚠️ Avoid for finance
💡 Intelli-Credit Recommendation Use a two-pass approach: Pass 1 = page-level extraction for structured financial data (BS, P&L, Cash Flow). Pass 2 = section-level chunking for qualitative text (MD&A, Risk Factors, Litigation) stored in a vector DB for RAG queries.

10 Practical Rules for Your Pipeline

🗂️
Always use TOC first Read the Table of Contents (usually pages 2–4) to build a page-range map before any extraction. This is free and saves all subsequent processing.
🎯
Target priority pages, scan the rest For a 350-page annual report, ~30 pages contain 80% of credit-relevant data. Deep-parse those, keyword-scan the remainder.
🔀
Never use one extractor for everything PyMuPDF for speed on text pages, Docling for financial tables, PaddleOCR only where there is no text layer. Mixed strategy beats any single tool.
📐
Chunk at page boundaries for finance Page-level chunking outperforms token-based chunking on financial RAG benchmarks. Don't split a Balance Sheet across chunks.
🔢
Always validate accounting identities After extraction, verify: Total Assets = Total Liabilities + Equity. PAT on P&L = PAT on Cash Flow. Mismatch = extraction error, not real.
⚠️
Prefer null over wrong If two verification passes disagree on a number, set it to null. A null in your CAM triggers manual review. A wrong number in your CAM causes bad lending decisions.
📝
Log which extractor processed which page "Page 201: Docling (multi-column table detected)". This helps debug extraction issues later without re-running everything.
🌐
Convert to Markdown as intermediate format Markdown preserves headings, tables, and bold text while remaining LLM-friendly. Use it as the canonical intermediate before JSON structuring.
⚡
Cache extracted text, don't re-parse PDFs Store extracted Markdown/JSON per PDF with its hash as key. If the PDF hasn't changed, serve from cache. Saves 90% of processing time on re-runs.
🇮🇳
Handle Indian number formatting explicitly Indian reports use lakh/crore notation (₹1,92,456 = ₹1.92 lakh, not 192,456). Parse Indian comma placement separately from Western. Add a normalizer step.
